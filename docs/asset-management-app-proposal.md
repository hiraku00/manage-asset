# ローカル資産管理アプリ 提案書・基本設計書

> 2026-07-11: 初期開発対象を暗号資産ウォレットに限定したため、最新の MVP 設計は [crypto-asset-tracker-design.md](crypto-asset-tracker-design.md) を参照する。本書は将来、銀行・証券等へ対象を広げる際の構想として残す。

- 文書版: 0.1
- 作成日: 2026-07-11
- 状態: 実装前提案

## 1. 結論

本アプリは、ブラウザだけで完結させず、PC 内でのみ動く「ローカル Web アプリ」とする。

```text
ブラウザ UI
    ↓ http://127.0.0.1 のみ
ローカル API / 取得オーケストレーター
    ├─ 銀行・証券コネクター
    ├─ 暗号資産取引所コネクター
    ├─ ウォレット・チェーンコネクター
    ├─ 手動 CSV / JSON 取込
    └─ 価格・為替コネクター
            ↓
    JSONL（原本）+ 任意の SQLite（表示用索引）
```

推奨方針は次のとおり。

1. 残高を取得する機能に限定し、売買・送金機能は持たせない。
2. API キーは必ず read-only とし、ブラウザや JSONL に保存しない。
3. 取得結果は上書きせず、取得ごとのスナップショットとして追記する。
4. 金融機関 API がなければ、正規のアグリゲーション API、公式 CSV、手入力の順で代替する。
5. ウォレットは DeBank 等の集約 API を第一候補、チェーン RPC / インデクサーを検証・補完用とする。
6. JSONL を監査可能な原本とし、検索が遅くなった場合だけ SQLite を再生成可能なキャッシュとして追加する。

この方式なら、外部ホスティングや常時稼働サーバーを必要とせず、ボタンを押した時点だけデータを取得できる。

## 2. 目的と対象範囲

### 2.1 目的

- 複数の銀行、証券会社、暗号資産取引所、オンチェーンウォレットの資産を一画面で確認する。
- 任意のタイミングで「現在値を取得」し、その時点の状態を履歴として残す。
- 保存済みの過去スナップショットを日付指定で閲覧する。
- 取得元、取得日時、評価価格、エラーを追跡できるようにする。
- 口座を任意のグループにまとめ、資産区分別にも集計する。

### 2.2 初期対象

- 銀行: 普通預金、定期預金、外貨預金など
- 証券: 現金、国内外株式、投資信託、債券等
- 暗号資産取引所: 現物、ステーキング等（取引所 API の対応範囲内）
- ウォレット: EVM 系を優先し、ネイティブ通貨、トークン、DeFi ポジションを対象とする
- 手動資産: 現金、不動産、未対応口座等
- 評価通貨: JPY を基準とし、USD 等を選択可能にする余地を残す

### 2.3 初期対象外

- 売買注文、送金、出金、署名
- 秘密鍵・シードフレーズの保管
- 税務申告額の確定
- 家計簿・支出分類の完全実装
- 複数ユーザー、外部公開、クラウド同期
- ブラウザ拡張として MetaMask 自体に接続する機能

「ウォレットを見る」だけなら公開アドレスで足りるため、MetaMask の接続許可や署名は不要である。MetaMask は資産の保管場所というより、参照対象アドレスを管理する入口として扱う。

## 3. 取得方式の提案

### 3.1 共通の優先順位

各口座は、次の優先順位で接続方式を決める。

| 優先 | 方式 | 長所 | 主な制約 |
|---:|---|---|---|
| 1 | 公式 API / OAuth | 安全性・安定性が比較的高い | 個人開発者に未開放、審査・費用がある場合がある |
| 2 | 読取専用 API キー | 実装しやすく残高取得に適する | キー管理、取引所ごとの差異 |
| 3 | 正規の集約 API | 多数の機関を統一形式で扱える | 契約、料金、トークン管理、利用条件 |
| 4 | 公式 CSV / PDF / JSON | 認証情報をアプリに渡さない | 手動操作、フォーマット変更、残高時点の差 |
| 5 | 手動入力 | どの資産にも対応できる | 入力ミス、更新忘れ |
| 非推奨 | 画面スクレイピング | 一見自動化しやすい | 規約、2FA、画面変更、ロック、資格情報漏えい |

スクレイピングは標準機能にしない。どうしても必要な場合も、利用規約の確認後に、ユーザー自身がダウンロードしたファイルをローカルで解析する方式を優先する。

### 3.2 銀行・証券

日本の銀行 API は、銀行が一般個人に直接自由な API キーを発行する形とは限らない。第三者として口座情報を継続取得するサービスには制度・契約上の要件があり、国内で電子決済等代行業を営む場合は登録制度がある。そのため、個人用ローカルアプリであっても「銀行ごとの API に直接つなげばよい」とは限らない。

候補は以下の順で評価する。

#### A. Moneytree LINK 等の正規アグリゲーション API

Moneytree LINK は OAuth 2.0 を採用し、銀行、クレジットカード、証券等を統一的に照会でき、更新要求用の仕組みも案内されている。対応機関は多いが、`client_id` / `client_secret` の発行や本番利用条件は事業者への確認が必要である。

採用判断:

- 個人開発・単一利用者でも本番契約可能か
- 初期費用、月額費用、最低利用数
- データの二次保存、保存期間、ローカル保存の可否
- 証券の保有数量・銘柄・取得価額がどこまで返るか
- 更新頻度、追加認証、障害時の扱い

費用や契約条件が目的に合わない場合は、初期版から外す。

#### B. 金融機関が提供する公式 API

対象金融機関ごとに個人向け利用可否を調査してコネクター化する。法人向け API、提携事業者専用 API、決済専用 API は、残高閲覧目的で使えない可能性がある。対応表には単なる「API あり」ではなく、認証方式、個人利用可否、取得項目、履歴期間を記録する。

#### C. 公式 CSV / 電子交付データ

API が使えない銀行・証券会社では、利用者が公式サイトから CSV をダウンロードし、画面へドラッグ＆ドロップする方式を標準代替とする。元ファイルのハッシュと取込日時を保存し、重複取込を防ぐ。

証券 PDF は抽出精度の問題があるため CSV を優先し、PDF は「要確認」付きの補助入力とする。

#### D. 手動残高

CSV もない場合は、日付、口座、通貨、数量、メモを手動登録する。履歴を壊さないよう訂正も追記イベントとして保存する。

### 3.3 暗号資産取引所

取引所ごとに read-only API キーを作成し、IP 制限を設定できる場合は localhost を実行する PC の外向き IP に限定する。出金・注文権限は付与しない。

コネクターは二層構造を推奨する。

- 共通層: CCXT の `fetchBalance`、`fetchPositions`、`fetchLedger` 等を利用
- 個別層: CCXT で不足する Earn、ステーキング、サブ口座等だけ取引所固有 API を利用

CCXT は多数の取引所 API を統一するが、機能対応は取引所ごとに `true`、`false`、`emulated` 等で異なる。したがって「CCXT 対応 = 全残高を取得可能」とはみなさず、取引所別の照合テストを行う。

初回導入時は、取引所画面の合計と以下を照合する。

- 現物の available / locked
- 証拠金・先物口座（必要な場合）
- Earn / ステーキング
- サブ口座
- 小額残高、上場廃止銘柄

### 3.4 MetaMask アドレス・オンチェーン資産

対象は「指定した公開アドレス」であり、秘密鍵やシードフレーズは一切要求しない。

#### 推奨する二段構成

1. **集約取得**: DeBank API 等で、そのアドレスの使用チェーン、トークン、プロトコルポジション、NFT、総評価額を取得する。
2. **検証・補完**: チェーン RPC またはインデクサーでネイティブ通貨、主要トークン、主要プロトコルを再確認する。

DeBank の公式 API には、全対応チェーンの総残高、使用チェーン、トークン一覧、プロトコルポジション、NFT、履歴等のエンドポイントが案内されている。ただし、価格未登録トークン、非対応チェーン、新規プロトコル、スパムトークン等は完全でない可能性があるため、以下を表示する。

- `source`: DeBank / RPC / 手動等
- `coverage`: complete / partial / unknown
- `verified`: 集約値と RPC 値の照合状態
- `spam_suspected`: スパム疑い
- `price_status`: priced / unpriced / stale

#### チェーン探索

- DeBank 等の `used_chain_list` を起点にする。
- ユーザーが明示指定したチェーンは残高ゼロでも照会する。
- EVM 以外（Bitcoin、Solana 等）はアドレス形式も取得 API も異なるため、別コネクターとして追加する。
- 同じ文字列の EVM アドレスを全チェーンに無制限照会すると、レート制限と待ち時間が増えるため、対応チェーン一覧と探索結果をキャッシュする。

#### DeBank 画面のスクレイピングについて

画面 HTML の解析より公式 API を優先する。画面の内容は表示用に加工され、DOM 変更や利用規約の影響を受ける。API 契約が難しい場合は、画面値を手動スナップショットとして入力できるようにする方が保守しやすい。

## 4. 「過去データ」の定義

過去日指定には、意味の異なる三機能がある。画面でも混同させない。

| モード | 意味 | 信頼度 |
|---|---|---|
| 保存済みスナップショット | その日に本アプリが実際に取得・保存した値 | 高 |
| 過去残高の再構築 | 取引履歴やチェーン状態から当時数量を計算 | 取得元次第 |
| 過去価格で再評価 | 保存数量または再構築数量 × 指定日の価格 | 価格元次第 |

### 4.1 原則

- 「今ボタンを押して、銀行の 3 か月前の正確な残高を得る」ことは、提供 API や明細期間によっては不可能。
- 初回利用以前の履歴は、取得元が返す取引明細、エクスポートファイル、オンチェーン履歴から可能な範囲だけ復元する。
- 初回利用後は定期的なスナップショットほど正確だが、本要件では手動取得を基本とする。
- 指定日にスナップショットがなければ、直前値を勝手に「当日値」とせず、`推定` と表示する。

### 4.2 オンチェーン

Ethereum JSON-RPC の `eth_getBalance` や `eth_call` はブロック番号を指定できるため、履歴状態を保持する RPC があれば過去ブロック時点を照会できる。ただし、トークン発見、DeFi ポジション評価、リベース、LP、借入等まで完全に復元するのは別問題である。

指定日を各チェーンの直前ブロックへ変換し、次を保存する。

- 対象日時とタイムゾーン
- chain ID
- block number / block hash
- 取得数量
- RPC 提供元
- 評価価格の基準日時

### 4.3 価格と為替

- 数量と価格を分離保存する。
- 取得時の評価額だけでなく、`quantity × unit_price × fx_rate` の各要素を残す。
- 暗号資産の過去価格は CoinGecko 等の履歴 API を候補とする。同公式 API は日付指定の過去データを提供し、日次値の基準時刻も明示している。
- 株価、投資信託基準価額、為替はライセンスと再配布条件を確認する。
- 価格がない資産はゼロ円扱いにせず「価格未取得」として合計の完全性を下げる。

## 5. システム構成

### 5.1 コンポーネント

| コンポーネント | 責務 |
|---|---|
| Browser UI | 取得指示、進捗、資産一覧、履歴、エラー表示 |
| Local API | UI 用 API、入力検証、localhost 制限 |
| Orchestrator | 複数コネクターの並列実行、タイムアウト、再試行 |
| Connector | 外部データを共通形式へ正規化 |
| Valuation | 価格・為替取得、評価額計算 |
| JSONL Store | 追記型の原本保存 |
| Read Model | 集計・検索用データ。初期はメモリ、必要時 SQLite |
| Secret Store | OS キーチェーン等で秘密情報を保存 |

### 5.2 ローカル Web アプリにする理由

- 外部 API の秘密情報をブラウザへ露出させない。
- ブラウザの CORS 制限をコネクター側で吸収できる。
- JSONL の排他制御、バックアップ、検証を一か所で行える。
- UI 技術とコネクター実装を分離できる。

Local API は既定で `127.0.0.1` のみに bind し、LAN へ公開しない。起動時にランダムなセッショントークンを UI に渡し、他サイトからの localhost 攻撃にも備える。

### 5.3 推奨技術（実装時に確定）

- Backend: Python + FastAPI
- Connector: Python ライブラリ、CCXT、各社 SDK / REST
- Frontend: React + TypeScript、または小規模なら server-rendered UI
- Chart: 軽量なチャートライブラリ
- Validation: JSON Schema / Pydantic
- Secrets: macOS Keychain（将来 Windows Credential Manager 等を抽象化）
- Distribution: 開発時は CLI 起動、安定後に単体アプリ化を検討

Python は金融・データ処理とコネクター実装に向き、FastAPI はローカル API とスキーマ生成を簡潔に保てる。UI を最小化したい場合は React を必須としない。

## 6. データ設計

### 6.1 保存原則

- JSONL は 1 行 1 レコード、UTF-8、追記専用とする。
- 金額・数量は浮動小数点数ではなく、10 進文字列で保存する。
- `observed_at`（資産の基準時刻）と `fetched_at`（実際の取得時刻）を分ける。
- 外部サービスの生レスポンスは、機密値を除去して別ファイルへ任意保存する。
- スキーマバージョンを全レコードに持たせる。
- 訂正や削除は原本を書き換えず、訂正イベントを追記する。

### 6.2 推奨ディレクトリ

```text
data/
  config/
    accounts.json                 # 秘密情報を含まない口座定義
    groups.json
  events/
    2026/07/snapshots.jsonl       # 正規化済みスナップショット
    2026/07/runs.jsonl            # 取得実行とエラー
    corrections.jsonl             # 手動訂正
  raw/
    2026/07/<run-id>/...          # 任意。秘匿化済み原本
  imports/
    manifests.jsonl               # CSV 等の取込履歴・ハッシュ
  cache/
    read-model.sqlite             # 削除・再生成可能
  backups/
```

`data/` 全体は Git 管理対象外とする。特に raw response と API キーはリポジトリへ入れない。

### 6.3 口座定義例

```json
{
  "schema_version": 1,
  "account_id": "acct_exchange_001",
  "display_name": "取引所A 現物",
  "institution_type": "crypto_exchange",
  "connector": "ccxt:exchange-a",
  "groups": ["personal", "long-term"],
  "base_currency": "JPY",
  "secret_ref": "keychain://manage-asset/acct_exchange_001",
  "enabled": true
}
```

### 6.4 スナップショット JSONL 例

```json
{"schema_version":1,"record_type":"position_snapshot","run_id":"run_01J...","snapshot_id":"snap_01J...","account_id":"wallet_001","group_ids":["personal"],"institution_type":"wallet","asset_class":"crypto","asset_id":"eip155:1/erc20:0x...","symbol":"USDC","name":"USD Coin","chain_id":"eip155:1","protocol_id":null,"quantity":"123.450000","unit_price":{"amount":"1.0001","currency":"USD","as_of":"2026-07-11T05:00:00Z","source":"coingecko"},"fx_rate":{"base":"USD","quote":"JPY","rate":"159.12","as_of":"2026-07-11T05:00:00Z","source":"..."},"value":{"amount":"19645.38","currency":"JPY"},"observed_at":"2026-07-11T05:01:02Z","fetched_at":"2026-07-11T05:01:09Z","source":{"connector":"debank","source_id":"..."},"quality":{"coverage":"partial","price_status":"priced","verified":false,"estimated":false},"raw_ref":"raw/2026/07/run_01J.../wallet_001.json"}
```

資産 ID は symbol だけにしない。同名トークンを区別するため、チェーンとコントラクトアドレスを含む CAIP 系の識別方法を採用する。銀行通貨は ISO 4217、株式は市場とティッカーまたは ISIN 等を組み合わせる。

### 6.5 取得実行レコード

1 回のボタン操作を `run` とし、口座ごとの成功・部分成功・失敗を記録する。

```json
{"schema_version":1,"record_type":"acquisition_run","run_id":"run_01J...","requested_at":"2026-07-11T05:00:00Z","requested_by":"local-user","mode":"current","status":"partial","accounts":{"succeeded":5,"partial":1,"failed":1},"completed_at":"2026-07-11T05:01:12Z"}
```

途中で 1 口座が失敗しても他口座の成功結果は保存する。UI では「全成功」と誤表示せず、部分成功と最終成功時刻を口座別に示す。

## 7. UI 設計

### 7.1 主要画面

#### ダッシュボード

- 総資産、負債、純資産
- グループ別、金融機関種別、資産クラス、通貨、チェーン別の内訳
- 前回比・期間推移
- 未評価資産額ではなく「未評価件数」と合計の不完全性
- 最終取得日時とデータ鮮度

#### 口座・資産一覧

- グループ → 種別 → 金融機関 → 口座 → 資産の階層
- 同時に、資産クラスやチェーンを軸に切り替え可能
- 数量、単価、為替、評価額、取得元、品質状態
- 重複候補の警告

#### データ取得

- 「すべて取得」と口座単位の取得
- current / historical / import のモード表示
- 進捗、レート制限、追加認証待ち、成功・失敗
- 実行後の増減差分と異常値確認

#### 履歴

- 日付または取得回を選択
- 保存済み / 再構築 / 再評価を明示
- スナップショット間の増減
- 価格変動と入出金による変動を、可能な範囲で分離

#### 設定

- 口座、公開ウォレットアドレス、グループ
- 接続テスト
- API キーは値を再表示せず、保存済みかだけ表示
- データ保存先、バックアップ、基準通貨

### 7.2 グループと分類

「銀行」「証券」「暗号資産」は金融機関種別または資産クラスであり、「個人」「法人」「長期」「生活防衛資金」等はユーザー定義グループである。両者を一つの階層に固定せず、別軸として持つ。

- `group_ids`: ユーザー定義、複数所属可
- `institution_type`: bank / brokerage / crypto_exchange / wallet / manual
- `asset_class`: cash / equity / fund / bond / crypto / nft / real_estate / liability / other
- `location`: 金融機関、口座、chain、protocol

この分離により、「個人グループの暗号資産」「全グループの銀行預金」の両方を簡単に集計できる。

## 8. 取得処理フロー

1. UI で対象口座と基準日時を選び、取得を開始する。
2. Local API が `run_id` を発行する。
3. Secret Store から必要な資格情報をプロセス内にだけ読み出す。
4. コネクターを制限付きで並列実行する。
5. 生データを検証し、共通 Position 形式へ正規化する。
6. 価格・為替を取得し、数量と分離して評価する。
7. 異常値・重複・欠落・未評価を判定する。
8. 一時ファイルへ書き、flush / fsync 後に JSONL へ安全に反映する。
9. 表示用索引を更新し、UI へ結果と警告を返す。

同じ `run_id + account_id + asset_id + observed_at` の再実行は冪等に扱い、二重計上を防止する。

## 9. セキュリティ・プライバシー

### 必須要件

- API キーは OS キーチェーンに保存し、JSON、環境設定画面、ログに平文保存しない。
- 暗号資産取引所キーは read-only、出金不可、注文不可、可能なら IP 制限付きにする。
- 銀行の ID・パスワードを独自保存しない。OAuth / 正規連携を優先する。
- 秘密鍵・シードフレーズは入力欄自体を作らない。
- Local API は loopback bind、認証付き、厳格な Origin チェックを行う。
- ログから Authorization header、Cookie、口座番号、氏名等をマスクする。
- `data/` と `.env` を `.gitignore` に含める。
- バックアップは暗号化し、復元テストを用意する。
- 依存ライブラリとコネクターのバージョンを固定し、更新時に回帰テストする。

### データ完全性

- 各 raw import の SHA-256 を保存する。
- JSONL 行に安定した ID を付ける。
- 将来必要なら日次ファイル単位のハッシュチェーンを加える。
- 数量、価格、為替、計算後評価額の整合性を読込時にも検証する。

公開ウォレットアドレス自体は公開情報でも、複数アドレスを同一人物の資産として束ねたデータは高いプライバシー性を持つため、銀行データと同等に保護する。

## 10. エラー・品質設計

コネクターは単なる成功 / 失敗ではなく次を返す。

- `success`: 想定範囲を取得
- `partial`: 一部口座、チェーン、商品、価格が欠落
- `stale`: 取得元が更新中で以前の値
- `auth_required`: OAuth 再認可や 2FA が必要
- `rate_limited`: 待機後に再試行可能
- `unsupported`: 商品やチェーンが非対応
- `failed`: 取得不能

品質フラグの例:

- 前回比が設定閾値を超える
- 同一資産が取引所 API と CSV の両方から重複
- 評価価格が 24 時間以上古い
- シンボルは同じだがコントラクトが異なる
- 集約 API と RPC の数量が不一致
- 負債を正の資産として返している可能性

## 11. JSONL 単独運用の評価

### 長所

- 人が確認しやすく、バックアップしやすい。
- 追記型で履歴と監査に向く。
- 特定 DB に依存しない。

### 注意点

- 日付・口座・資産をまたぐ集計は件数増加とともに遅くなる。
- 途中書込み、同時書込み、重複、スキーマ移行を自前で扱う必要がある。
- ブラウザから直接読むだけでは秘密情報を使う取得処理や安全な更新が難しい。

### 改善案

JSONL を原本のまま残し、SQLite を検索専用の派生データにする。SQLite を失っても JSONL から再構築できるため、可搬性と表示速度を両立できる。初期データ量が小さければ SQLite は後回しでもよいが、境界を先に定義しておく。

## 12. 段階的な開発案

### Phase 0: 対象口座の棚卸し

- 実際に使う銀行、証券会社、取引所、ウォレット、チェーンを一覧化
- 各社の API / CSV / 商品範囲 / 履歴期間 / 利用条件を調査
- 正解となる画面残高のサンプルを用意

成果物: 接続可否マトリクス、データ辞書、受入基準

### Phase 1: ローカル MVP

- 口座・グループ設定
- 手動入力と CSV 取込
- 1 取引所の read-only API
- 1 EVM アドレスの DeBank 等による取得
- JSONL スナップショット
- 現在資産、分類別集計、履歴表示

この段階では銀行の自動連携を必須にしない。先に共通データモデルと履歴の正しさを固める。

### Phase 2: 連携拡大

- 対象取引所の追加
- RPC 検証と複数チェーン対応
- 証券 CSV アダプター
- 価格・為替履歴
- SQLite read model、差分・品質警告

### Phase 3: 銀行・証券 API

- 契約条件が合えば Moneytree LINK 等を接続
- OAuth 再認可、追加認証、更新要求、障害対応
- 取引履歴からの過去残高再構築

### Phase 4: 運用品質

- 暗号化バックアップと復元
- コネクター回帰テスト
- スキーマ移行ツール
- 配布形式の改善

## 13. 受入基準案

- 取得ボタンから、対象口座ごとの成否と最終取得時刻を確認できる。
- 1 口座が失敗しても、他口座の成功データが失われない。
- 同じ取得を再実行しても二重計上されない。
- 数量、価格、為替、評価額、取得元、基準時刻を追跡できる。
- 保存済み過去スナップショットを日付で選べる。
- 推定値、未評価、部分取得を確定値と区別できる。
- グループ、金融機関種別、資産クラス、通貨、チェーンで集計できる。
- API キー、銀行資格情報、シードフレーズが JSONL やログに残らない。
- JSONL だけから表示用データを再構築できる。
- 取引所・ウォレットの表示残高と、定めた許容差内で一致する。

## 14. 実装前に決める事項

優先度の高い順に、次を確定する。

1. 対象金融機関・取引所名と、各口座の商品種別
2. 対象ウォレットアドレス、EVM 以外を含む対象チェーン
3. JPY 以外の基準通貨、評価時刻（日末か取得時点か）
4. NFT、DeFi、先物、ステーキング、負債を初期版に含めるか
5. Moneytree LINK 等の有料・契約型サービスを許容するか
6. raw response を保存するか、保存する場合の暗号化方針
7. 手動取得だけでよいか、将来のスケジュール取得も見込むか
8. macOS 専用でよいか、Windows 等にも対応するか

## 15. 主なリスクと対策

| リスク | 影響 | 対策 |
|---|---|---|
| API / CSV 仕様変更 | 取得停止・誤集計 | コネクター分離、fixture テスト、バージョン記録 |
| API 対象外の商品 | 資産の過少表示 | coverage 表示、画面残高との照合、手動補正 |
| 過去残高を取得できない | 履歴の欠落 | 早期からスナップショット保存、CSV 明細取込 |
| 無価値・スパムトークン | 総額の誤表示 | allow/deny、価格品質、非表示フラグ |
| 二重計上 | 総資産の過大表示 | 一意 ID、重複候補、保管場所の移動を区別 |
| API キー漏えい | 資産・個人情報リスク | read-only、Keychain、ログ秘匿、Git 除外 |
| 外貨・価格時点の不一致 | 比較不能 | 数量・単価・FX・時刻を分離保存 |
| JSONL 肥大化・破損 | 表示低速・履歴消失 | 月次分割、原子的追記、バックアップ、SQLite 索引 |

## 16. 提案の要点

最も重要な改善点は、単に「各サービスの現在残高を JSONL に保存する」だけでなく、以下を最初からデータモデルに含めることである。

- 取得日時と残高基準日時の分離
- 数量・価格・為替・評価額の分離
- 取得元とデータ品質
- 部分成功と未対応範囲
- 保存済み値、再構築値、推定値の区別
- グループと資産分類の分離

これにより、対応先を増やしても集計の意味が崩れにくく、後から誤差や欠落の原因を確認できる。
