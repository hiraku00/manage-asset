# 暗号資産ポートフォリオ UI/UX監査・改修仕様書

- 文書版: 1.0
- 作成日: 2026-07-12
- 対象実装: `static/index.html` の全画面
- 目的: デジタル庁デザインシステム（DADS）の考え方を踏まえ、個人が短時間で資産状況を理解し、安全に更新できるUIへ整理する
- 本書の位置付け: 実装前の監査結果および改修要件。既存コードの見た目を肯定するものではない

## 1. 結論

現状は必要な機能が一画面に揃っている一方、情報の優先順位、用語、操作の段階、状態表示が十分に統一されていない。特に次の点が認知負荷を高めている。

1. 「資産を確認する」と「データを取り込む」が同程度の強さで見える。
2. 総資産、為替、更新時刻の意味と基準日時が一貫していない。
3. 取引所更新とウォレット更新で、操作順・ボタンの重要度・完了状態が異なる。
4. 表示名、内部用語、API用語が混在する。
5. 表、グラフ、ステータスに同じ情報が重複し、重要な異常が埋もれる。
6. 空状態、処理中、失敗、部分成功、未評価の区別が弱い。

改修後は「見る」「更新する」「設定する」の3責務を明確に分ける。トップ画面は判断に必要な情報だけを置き、通常時の説明文を減らす。警告や技術詳細は問題がある時だけ段階的に表示する。

## 2. 準拠方針

DADSを外観の模倣として扱わず、次の原則を設計判断に使う。

- アクセシビリティを後付けせず、情報設計・文言・操作順から組み込む。
- 見出し階層を飛ばさず、見出しだけでページ構造を把握できるようにする。
- ボタンは操作結果が分かる具体的なラベルにし、重要度を視覚だけに依存させない。
- フォームはプレースホルダーをラベル代わりにせず、入力例と説明を別に示す。
- データテーブルは列を増やしすぎず、セルの関係が単純な構造にする。
- 色だけで正常・注意・失敗を表現せず、短いテキストと記号を併用する。
- フォーカス表示、キーボード順、44px程度の操作領域、十分なコントラストを維持する。
- 情報を隠しすぎず、主情報・補足情報・詳細情報を段階的に開示する。

## 3. 想定ユーザーと主要タスク

個人利用でも、画面は「開発者が内部構造を知っている」ことを前提にしない。

主要タスクは次の順に扱う。

1. 現在の総資産を確認する。
2. 主な資産と保管場所を確認する。
3. 前回からの増減を確認する。
4. データがいつ取得されたか確認する。
5. ウォレットまたは取引所を更新する。
6. 取得失敗や未評価資産を調べる。
7. ウォレット・取引所の接続設定を管理する。

通常利用の中心は1〜4であり、更新と設定をファーストビューの主役にしない。

## 4. 推奨する全体情報設計

### 4.1 グローバルナビゲーション

| 現在 | 推奨 | 理由 |
|---|---|---|
| 資産概要 | 資産概要 | 維持 |
| 保管場所 | 保管場所 | 維持 |
| データ更新 | データ更新 | 維持 |
| 設定 | 設定 | 維持 |
| 右上の最終更新 | 削除 | 資産概要内と重複し、他画面では意味が曖昧 |
| データを更新 | 更新画面を開く | 遷移であることを明示 |

現在地は下線だけでなく、`aria-current="page"`相当の状態を付ける。タブ風ボタンではなく、ページナビゲーションとして一貫させる。

### 4.2 画面責務

- 資産概要: 読み取り専用。総額、推移、配分、資産一覧。
- 保管場所: 読み取り専用。ウォレット・DeFi・取引所の評価額と鮮度。
- データ更新: 取得・確認・保存。設定変更は行わない。
- 設定: 接続先とウォレットを管理。残高取得は接続確認に限定する。

## 5. 共通レイアウト仕様

### 5.1 コンテンツ幅

- 最大幅は1440px程度を維持する。
- デスクトップ左右余白は24〜32px、モバイルは16pxを基準にする。
- カード内余白は24px、関連項目間16px、セクション間24〜32pxで統一する。
- `!important`や`:has()`で場当たり的に配置を補正せず、明示的なDOM構造とクラスを用いる。

### 5.2 見出し

- 各画面に`h1`は1つ。
- 主要カードは`h2`、カード内の小区分は`h3`。
- 説明文は見出し直下に必要な場合だけ置く。
- 「〜を確認します」のような自明な説明は削除する。

### 5.3 数値

- USDを主表示、JPYを補助表示とする。
- 通貨記号と桁区切りを統一する。
- 数量は最大8桁を基本とし、極小値は指数表記または`<0.00000001`。
- 評価額と数量を混同しない。
- 為替レートは値と取得日時をセットで示す。

### 5.4 文言

- 「取得」「更新」「保存」を使い分ける。
  - 取得: 外部サービスからデータを読む。
  - 更新: 最新データを取得し直す操作全体。
  - 保存: 確認済みスナップショットをJSONLへ確定する。
- `unavailable`、`spot`、`unified`等を利用者向け表示に出さない。
- APIエラーは「何が起きたか」「利用者が次に何をするか」を示す。

## 6. 資産概要画面

### 6.1 ページ見出し

推奨:

```text
資産概要
ウォレットと取引所の最新データを合算しています
```

説明は1行以内。DeFiを別の取得元のように列挙せず、保管場所として扱う。

### 6.2 総資産サマリー

表示は3列に限定する。

| 列 | 主表示 | 補助表示 |
|---|---|---|
| 総資産 | `$234,611.72` | `¥37,976,599`、前回比 |
| 換算レート | `USD/JPY 161.87` | `取得日時: 2026/07/12 16:43` |
| 最終更新 | `2026/07/12 16:43` | 古い取得元がある時だけ注意 |

要件:

- 3列の上端を揃える。
- 「総資産」だけを最大サイズにする。
- 為替の取得日時を太字値へ含めない。
- 「保管場所13件」「最古の取得」はトップから削除する。
- 更新時刻は各取得元で異なるため、最新時刻だけで完全更新と誤認させない。古い取得元がある場合のみ`1件が24時間以上前`と表示する。
- 前回比は比較基準を`前回保存比`と明示し、比較データがなければ行ごと非表示にする。

### 6.3 資産推移

現状評価:

- 保存済みデータのみを使う方針は妥当。
- 点の値とY軸はあるが、期間選択と不完全な日付の説明がない。

改修要件:

- 見出し右の文言は`保存したスナップショット`にする。
- 期間切替は履歴が十分に増えてから追加し、現時点では不要。
- X軸は年月日、Y軸はUSD。JPYを同じ軸に重ねない。
- 各点のツールチップに日付、USD、JPY、使用レートを表示する。
- 同日に全取得元が揃っていない場合は`一部データ`と明示する。
- 1点のみならグラフを描かず、`推移を見るには異なる日付のデータが2件以上必要です`と表示する。

### 6.4 資産配分

- 上位5資産とその他に統一する。見出しと実装の件数を一致させる。
- ドーナツ中央は総資産と同額であることを検証する。
- 凡例は資産、数量、USD、JPY、比率を同じ行構造で示す。
- 色だけに依存せず、資産名と比率を必ず表示する。
- その他はクリックまたは詳細リンクで内訳へ移動できるようにする。
- `総額と取得できた資産内訳は一致しています`は正常時には表示しない。差異がある時だけ警告を出す。

### 6.5 保有資産

推奨列:

| 列 | 表示 |
|---|---|
| 資産 | シンボル。必要時のみ名称 |
| 合計数量 | 全取得元の数量合計 |
| 評価額 | USDを主、JPYを小さく併記 |
| 比率 | 数値とバー |
| 主な保管場所 | 上位3件＋`その他N件` |

改善点:

- 保管場所は金額寄与順に並べる。現在の検出順では「上位3件」にならない。
- `その他N件`はツールチップまたは展開で全件確認できるようにする。
- `$1未満も表示`はチェックボックスより、`1 USD未満を表示`のスイッチまたは開閉ボタンが分かりやすい。
- 非表示件数と合計額は一覧下部へ簡潔に表示する。
- 未評価資産は金額を0として扱わず、`評価なし`と明示する。
- 同一シンボルでも異なるコントラクトを無条件に合算しない。

## 7. 保管場所画面

### 7.1 目的

「どこに、いくらあり、いつ取得したか」を確認する画面とする。資産概要と同じ情報を繰り返すだけにしない。

### 7.2 推奨表示

デスクトップは次の簡潔なテーブルとする。

| 保管場所 | 種別 | 評価額 | 構成比 | 最終取得 | 状態 |
|---|---|---:|---:|---|---|![alt text](image.png)
| Main Wallet | ウォレット | $… / ¥… | …% | 7/12 16:43 | 最新 |
| Aave V3 | DeFi | $… / ¥… | …% | 7/12 16:40 | 最新 |
| Binance | 取引所 | $… / ¥… | …% | 7/12 16:39 | 最新 |

要件:

- ウォレット単位だけでなく、DeFiと取引所も同じ粒度で比較できるようにする。
- 状態は`最新`を常時強調しない。注意・失敗だけを目立たせる。
- `一部未評価`には未評価資産名と影響を示す。
- モバイルで時刻や状態を単純に非表示にせず、2段レイアウトへ再配置する。
- 行を選択すると、その保管場所の資産内訳を表示する。

## 8. データ更新画面

### 8.1 画面構成

取引所とウォレットを同一ページに置く場合も、操作ステップを統一する。

```text
1. 更新対象を選ぶ
2. 最新データを取得する
3. 内容を確認する
4. 保存する
```

### 8.2 取引所更新

現状の`残高を確認`は意味が弱い。推奨:

- 主ボタン: `最新残高を取得`
- 確認後の主ボタン: `この内容を保存`
- 戻る操作: `取得結果を破棄`

プレビューには次を表示する。

- 取引所名
- 取得日時
- 合計評価額（USD・JPY）
- 資産件数
- 未評価件数
- 1 USD以上の明細
- 1 USD未満は折りたたみ

取得と保存でAPIを再度呼ばない。保存時は確認したプレビューと同じ内容を確定し、価格変動による差を防ぐ。

### 8.3 ウォレット更新

現状の長い一文による手順説明は分割する。

推奨構成:

- `全ウォレットを更新`を主要操作として上部に置く。
- 手動取込は`個別に取り込む`という副セクションへ置く。
- 記録日は`保存日`または`記録日`とし、過去残高を取得する日ではないことを近接表示する。
- `DeBankで開く`、`HTMLを読み込む`、`保存`を順番に有効化する。
- 自動取得中は対象件数、現在位置、成功・失敗数を表示する。
- ボタンをdisabledにするだけでなく、処理中であることをテキストで示す。

### 8.4 状態文言

| 状態 | 推奨文言 |
|---|---|
| 初期 | `更新対象を選択してください` |
| 取得中 | `Binanceから最新残高を取得しています` |
| 確認待ち | `取得結果を確認し、保存してください` |
| 保存済み | `2026/07/12 16:43のデータを保存しました` |
| 部分成功 | `12件中11件を保存しました。1件を確認してください` |
| 失敗 | `取得できませんでした。API権限を確認してください` |

## 9. 設定画面

### 9.1 情報順序

推奨順:

1. 接続済み取引所
2. 取引所を追加
3. 登録済みウォレット
4. ウォレットを追加

既存設定を確認する頻度が、新規追加より高いためである。

### 9.2 取引所設定

- `取引所`と`表示名`に常時ラベルを表示する。
- プレースホルダー`例: Binance Main`だけに入力意図を依存しない。
- `接続先を追加`は`取引所を追加`へ変更する。
- 認証情報保存と接続確認を一つの長いボタン名にせず、`認証情報を保存`後に接続結果を表示する。
- 読取専用要件は短い注意として入力欄の直前に置く。
- API Secretは再表示しない。保存済み状態と更新操作を分ける。
- 接続済み一覧に`認証情報を更新`、`接続確認`、`削除`を提供する。

### 9.3 ウォレット設定

- 初期値としてゼロアドレスを入力しない。空欄から開始する。
- 表形式の入力より、1ウォレット1カードまたは行フォームを用いる。
- 表示名、公開アドレス、有効・無効、削除を同じ行で扱う。
- アドレス形式エラーは保存後ではなく入力欄近傍へ表示する。
- 削除は確認ダイアログを出し、保存済み履歴への影響を説明する。
- `ウォレット設定を保存`ではなく、変更がある時だけ`変更を保存`を有効にする。

## 10. フォームとボタンの共通仕様

### 10.1 ボタン階層

- 1セクションに主ボタンは原則1つ。
- 塗りボタン: 次へ進む、取得する、保存する。
- アウトライン: 補助操作、外部ページを開く。
- テキスト: キャンセル、詳細、破棄。
- 同じ行に主ボタンを複数並べない。

### 10.2 disabled

disabledだけで操作不能理由を伝えない。ボタン近くに`先に最新残高を取得してください`等を表示する。可能なら操作可能状態を保ち、押下時に必要な案内を出す。

### 10.3 入力

- ラベルを常時表示する。
- 必須・任意を明記する。
- 入力例はサポートテキストにする。
- エラーは入力欄の直下に表示し、ページ上部だけへ集約しない。
- API Keyのコピー＆ペーストを妨げない。

## 11. テーブル

- `thead`、`tbody`、`th scope="col"`を用いる。
- 行見出しとなる資産名は`th scope="row"`を検討する。
- 数値列は右揃え、テキスト列は左揃え。
- モバイルで横幅が足りない場合は、横スクロールかカード表示に切り替える。重要列を無断で消さない。
- 空テーブルは枠だけを表示せず、次の操作を案内する。
- 並び順は評価額降順を初期値にし、列見出しで変更できる場合は状態を明示する。

## 12. グラフ

- SVGにはタイトルと説明を付ける。
- 同内容のテーブルまたは一覧を近接配置する。
- 色だけで系列を識別させない。
- 固定色は資産シンボルではなく正規化済み資産IDへ割り当てる。
- 円グラフの細片を増やさず、上位5件＋その他とする。
- 値が0、未評価、負債を視覚的に区別する。
- 動きは原則不要。追加する場合は`prefers-reduced-motion`へ対応する。

## 13. 状態・エラー・データ品質

### 13.1 状態モデル

全取得処理で次を共通化する。

```text
idle → loading → preview → saving → success
                    ↘ error
```

### 13.2 エラーの粒度

- ネットワーク接続
- TLS証明書
- API認証
- API権限
- レート制限
- 外部画面構造変更
- 価格未取得
- 一部取得成功

内部例外をそのまま出さず、詳細ログには原因、画面には解決手順を示す。APIキー、署名、Secretをログへ書かない。

### 13.3 正常表示

正常であることを大きな通知で繰り返さない。`総額と内訳は一致しています`等は削除し、異常時だけ警告する。

## 14. レスポンシブ

### デスクトップ

- 総資産サマリーは3列。
- 推移と配分は1:1。
- 保有資産はテーブル。

### タブレット

- 総資産を1行目、換算レート・最終更新を2行目にする。
- チャートは縦積みを許容する。

### モバイル

- ナビゲーションは横スクロールではなく、必要ならメニュー化する。
- サマリーは縦積み。
- テーブルは資産、評価額、比率を優先したカード表示にする。
- 保管場所、JPY、取得日時は詳細として同じカード内に残す。
- 更新フォームのボタンは縦並び・全幅。

## 15. アクセシビリティ受入条件

- キーボードのみで全操作を完了できる。
- フォーカス順と視覚順が一致する。
- フォーカスリングが背景に対して明確である。
- テキストと背景のコントラスト比4.5:1以上を目標にする。
- 色なしでも正常・注意・失敗を判別できる。
- 200%拡大で内容・操作が欠落しない。
- 320 CSS px幅で主要タスクを完了できる。
- 見出し階層が連続している。
- 動的更新は必要な範囲だけ通知し、`aria-live`で長文や頻繁な進捗を読み上げ続けない。
- グラフの内容を一覧でも確認できる。

## 16. 優先度付き改修バックログ

### P0: 誤認・操作失敗を防ぐ

1. サマリーを総資産・換算レート・最終更新の3列で安定化する。
2. 上位5資産表記とドーナツの実装件数を一致させる。
3. 取引所のプレビュー内容をそのまま保存し、再取得による差をなくす。
4. 保有資産の保管場所を金額順にする。
5. 未評価と0 USDを区別する。
6. ゼロアドレスを新規ウォレットの初期値にしない。

### P1: 認知負荷を下げる

1. 通常時の冗長な説明・整合性メッセージを削除する。
2. 全ウォレット更新をウォレット更新の主操作にする。
3. 設定画面を一覧優先に並べ替える。
4. 状態文言とボタンラベルを統一する。
5. 保管場所詳細を追加する。
6. USD・JPY・基準日時の表示規則を共通化する。

### P2: アクセシビリティと保守性

1. セマンティックなナビゲーション・テーブル・見出しへ修正する。
2. インラインHTML/CSS/JavaScriptを分割する。
3. デザイントークンをDADSに近い命名・尺度へ整理する。
4. 画面幅別のビジュアル回帰テストを追加する。
5. キーボード、スクリーンリーダー、200%拡大の確認手順を追加する。

## 17. 実装時の完了条件

- 主要4画面の全項目が本書の責務に沿っている。
- 初見の利用者が、総資産・主な資産・取得時刻を10秒以内に把握できる。
- ウォレット一括更新と取引所更新を、説明書なしで完了できる。
- 正常時には警告が表示されず、異常時には原因と次の操作が分かる。
- 資産合計、保管場所合計、配分グラフの集計対象が一致する。
- デスクトップ、タブレット、モバイルで情報が欠落しない。
- DADSおよびWCAGの観点で、見出し、フォーム、ボタン、テーブル、色、キーボード操作を確認済みである。
