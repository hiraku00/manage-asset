## 1. 結論

最初のバージョンは、EVM 形式のウォレットアドレスを登録し、利用者が DeBank ページ上で手動コピーした HTML をクリップボードから取り込み、画面に表示されているチェーン・資産を JSONL に保存する、小さなローカル Web アプリとする。DeBank 公式 OpenAPI は任意の代替取得手段として残す。

```text
Brave で開くローカル画面
  http://127.0.0.1:PORT
          │
          │「クリップボードから取り込む」
          ▼
ローカルアプリ
  ├─ コピー済み HTML をテキストとして受け取る
  ├─ アドレス・チェーン・資産情報を解析
  ├─ 取得結果を共通形式に変換
  └─ JSONL にスナップショットを追記
          │
          ▼
  総資産 / アドレス別 / チェーン別 / 資産別に表示
```

ブラウザは DeBank の表示と、利用者自身による HTML コピーに使用する。アプリは拡張ボタンの自動押下、座標クリック、バックグラウンド監視を行わない。クリップボードの読込みは、利用者がローカル画面の取込ボタンを押した時だけ実行する。

## 2. DeBank 取得方式

当初の案は、起動済み Brave で DeBank の各プロフィールページを開き、拡張機能やスクリプトで表示内容を順番に抽出するものだった。

Playwrightによるウォレット個別更新と全ウォレット一括取得を提供する。個別更新は選択したウォレットを取得・保存し、自動方式は登録済みの有効ウォレットを順番に取得する。画面上の手動HTML取込は現行UIでは提供しない。

なお、個人が自分で DeBank ページを開いて目視確認する機能は残す。アプリ上の「DeBank で確認」を押すと、既定ブラウザで次のページを開く。

```text
https://debank.com/profile/<wallet-address>
```

既定ブラウザが Brave で、すでに起動していれば、通常はその Brave に新しいタブが開く。アプリはそのタブからデータを抽出しない。

### 2.1 手動 HTML 取込の境界

```text
アプリ: DeBank プロフィールを Brave で開く
利用者: 表示完了を確認する
利用者: 既存の拡張ボタンを手動で押す
拡張: HTML をクリップボードへコピーする
利用者: ローカル画面で「クリップボードから取り込む」を押す
アプリ: HTML を解析して確認画面を出す
利用者: 内容を確認して保存する
```

アプリは次を行わない。

- Brave 上の拡張ボタンを座標指定で自動押下する
- DeBank ページへ content script を注入する
- DeBank の DOM や通信を直接読み取る
- クリップボードを常時監視する
- ページを自動巡回する

この境界により、アプリの責務を「利用者が明示的に提供した HTML のローカル解析」に限定する。

## 3. MVP の目的

複雑な会計や取引分析ではなく、次の質問にすぐ答えられることを目的とする。

- 全ウォレットを合計すると、およそいくら保有しているか
- どのアドレスにいくらあるか
- Ethereum、Arbitrum、Base 等の各チェーンにいくらあるか
- どのトークンをどれだけ保有しているか
- DeFi に預けている資産や借入がいくらあるか
- 前回取得時からどれくらい増減したか
- 最後にいつ取得したデータか

## 4. MVP の対象範囲

### 4.1 対象

- 複数の EVM ウォレットアドレス
- DeBank が対応する複数チェーン
- ウォレット内のネイティブ通貨・ERC-20 等
- DeBank が認識する DeFi プロトコルポジション
- プロトコル内の資産と負債
- USD 評価額
- 任意の USD/JPY レートによる参考 JPY 評価額
- ボタンを押した時点のスナップショット保存
- 保存済みスナップショットの履歴表示

### 4.2 初期対象外

- 銀行、証券会社、暗号資産取引所のログイン口座
- 売買、送金、承認、署名
- 秘密鍵、シードフレーズ、MetaMask 接続
- NFT の詳細表示
- 損益計算、取得原価、税務計算
- 取引履歴の完全な取込
- 指定した過去日時のオンチェーン残高再構築
- EVM 以外の Bitcoin、Solana 等
- 自動スケジュール取得
- 外部公開やクラウド同期

NFT、履歴、EVM 以外は、MVP が安定してから必要なものだけ追加する。

## 5. 利用イメージ

### 5.1 初期設定

1. ローカルアプリを起動する。
2. Brave でローカル画面を開く。
3. ウォレット名と公開アドレスを登録する。
4. 必要なら「個人」「保管用」「DeFi」等のグループを付ける。
5. 公式 API も使う場合だけ DeBank Cloud AccessKey を設定する。

登録に必要なのは公開アドレスだけである。秘密鍵やウォレット接続は不要とする。

### 5.2 最新データ取得

1. ウォレット詳細で「DeBank で確認」を押す。
2. Brave に開いたページの表示完了を確認する。
3. 利用者が既存の拡張ボタンを押し、HTML をコピーする。
4. ローカル画面の「クリップボードから取り込む」を押す。
5. アプリがアドレス、総額、チェーン、資産等を解析する。
6. 利用者がプレビューと警告を確認して保存する。
7. 必要な場合だけ次のアドレスで同じ操作を行う。

HTML の解析に失敗した場合は JSONL へ不完全な確定データを保存せず、検出できた項目と失敗理由をプレビューへ表示する。

### 5.3 過去データを見る

- 取得履歴から日時を選ぶ。
- その取得回の総額、アドレス別、チェーン別内訳を表示する。
- 指定日の取得データがなければ、直近値を指定日値として見せない。

### 6.1 ダッシュボード

最初の画面に必要な情報だけをまとめる。

- 総資産 USD
- 参考総資産 JPY
- 前回取得比

- 取得成功アドレス数 / 登録アドレス数
- アドレス別内訳
- チェーン別内訳
- 資産上位一覧
- DeFi ポジション一覧
- 未評価トークン数

### 6.2 表示切替

同じデータを次の単位で切り替える。

| 表示 | 内容 |
|---|---|
| アドレス別 | ウォレット名、アドレス、合計額、前回比 |
| チェーン別 | チェーン名、全アドレス合計、構成比 |
| 資産別 | トークン名、数量、単価、評価額、保管場所 |
| DeFi 別 | プロトコル、チェーン、資産、負債、純資産 |
| 履歴 | 取得日時ごとの総額と内訳 |

### 6.3 ウォレット詳細

- 表示名
- 公開アドレス
- グループ
- 合計評価額
- チェーン別残高
- ウォレット内トークン
- DeFi ポジション
- 取得時刻と取得状態
- 「このアドレスだけ更新」
- 「DeBank で確認」
- 「クリップボードから取り込む」
- HTML 解析プレビュー

### 6.4 設定

- アドレス追加・編集・無効化
- グループ設定
- DeBank AccessKey の登録・更新（公式 API を使う場合のみ）
- 基準通貨 USD / JPY
- JSONL 保存先
- 接続テスト

## 7. システム構成

### 7.1 最小構成

| 部品 | 役割 |
|---|---|
| Browser UI | 資産表示、更新ボタン、設定 |
| Local API | UI と取得処理の窓口 |
| HTML Importer | クリップボード HTML の検証、解析、正規化 |
| DeBank Connector | 任意。公式 OpenAPI 呼出しと正規化 |
| Snapshot Store | JSONL への追記・読込 |
| Valuation | USD 集計、任意の JPY 換算 |

初期版ではデータベースを使わない。JSONL を読み込んでメモリ上で集計する。データが増えて表示が遅くなった時だけ SQLite の追加を検討する。

### 7.2 推奨技術

- Python 3
- FastAPI または同程度の軽量ローカル Web フレームワーク
- HTML + 少量の JavaScript
- 必要になった場合だけ小さなフロントエンドライブラリ
- JSONL
- macOS Keychain または環境変数による AccessKey 管理

個人用 MVP では React 等の大きな構成を必須にしない。表、絞り込み、簡単な推移チャートが実現できれば十分とする。

### 7.3 ローカル限定

- `127.0.0.1` のみに bind する。
- LAN の別端末からアクセスできないようにする。
- 外部ホスティングしない。
- データファイルを Git に含めない。

## 8. DeBank 取得設計

### 8.1 手動 HTML 取込モード

MVP の標準取得方式とする。ローカル画面で取込ボタンを押した時だけクリップボードを読み、次を順に行う。

1. クリップボードが HTML または HTML を含むテキストか確認する。
2. 最大サイズを確認し、異常に大きいデータを拒否する。
3. HTML を実行せず、パーサーで静的な文書として読み込む。
4. 対象アドレスまたはプロフィール URL を検出する。
5. 登録アドレスと一致するか確認する。
6. 総資産、チェーン、トークン、DeFi ポジションを抽出する。
7. 数値と通貨単位を正規化する。
8. 抽出結果と未解析項目をプレビューする。
9. 利用者の保存操作後に JSONL へ追記する。

HTML 内の JavaScript、イベント属性、外部画像・外部 URL は実行・読込しない。プレビューへ HTML をそのまま挿入せず、抽出後のプレーンテキストだけをエスケープして表示する。

#### 抽出項目

| 項目 | 必須 | 失敗時 |
|---|---:|---|
| ウォレットアドレス | 必須 | 保存不可 |
| 取得時刻 | 必須 | アプリ側時刻を使用し、その旨を記録 |
| 総資産 USD | 必須 | 保存前に手動確認を要求 |
| チェーン名・チェーン評価額 | 推奨 | 部分取得として保存可能 |
| トークン名・数量・評価額 | 推奨 | 部分取得として保存可能 |
| DeFi プロトコル・純資産 | 推奨 | 部分取得として保存可能 |
| 負債 | 推奨 | 未取得警告を表示 |

#### HTML 変化への対応

DeBank の画面構造は変更され得るため、CSS class 名だけに依存しない。

- 見出し、ラベル、データ属性、URL、階層関係を組み合わせて判定する。
- 英語表記を初期対象とし、日本語等が表示される場合はラベル辞書を追加する。
- 金額の `$`、桁区切り、`< $0.01` 等を個別に処理する。
- パーサーバージョンをスナップショットへ保存する。
- 個人情報を除去した HTML fixture で回帰テストする。
- 必須項目を検出できなければ、推測で保存しない。

#### クリップボード UX

- 取込前に `HTMLをコピーしてから押してください` と表示する。
- 読み取った内容の先頭や生 HTML は画面へ表示しない。
- 成功後もクリップボードをアプリが勝手に消去しない。
- 同じ HTML の SHA-256 を記録し、同一内容の重複取込を警告する。
- 自動保存せず、必ず確認画面を挟む。

### 8.2 公式 API モード（任意）

#### 使用候補エンドポイント

MVP ではアドレスごとに次を使う。

| API | 用途 | MVP |
|---|---|---:|
| `/v1/user/total_balance` | 全チェーン総額とチェーン別評価額 | 必須 |
| `/v1/user/used_chain_list` | そのアドレスで使用されたチェーン | 必須 |
| `/v1/user/all_token_list` | 全対応チェーンのトークン残高 | 必須 |
| `/v1/user/all_complex_protocol_list` | DeFi の詳細ポジション | 必須 |
| `/v1/user/complex_app_list` | Hyperliquid 等のアプリポジション | 任意 |
| `/v1/user/all_nft_list` | NFT | MVP 後 |
| `/v1/user/all_history_list` | 取引履歴 | MVP 後 |

API 仕様と消費 unit は実装開始時に再確認する。DeBank Cloud は AccessKey と compute unit 方式を案内しており、価格・無料試用・各 API の unit 消費は変更され得る。

#### アドレス単位の処理順

```text
1. total_balance
2. used_chain_list
3. all_token_list
4. all_complex_protocol_list
5. complex_app_list（設定が有効な場合）
6. 正規化
7. 合計照合
8. JSONL 保存
```

アドレスは最初は一つずつ処理する。高速化より、API 制限を避け、どのアドレスで失敗したか分かりやすくすることを優先する。

#### 待機と再試行

- タイムアウト: API ごとに設定
- 一時エラー: 最大 2 回程度、間隔を増やして再試行
- 429: `Retry-After` があれば従う
- 401 / 403: AccessKey エラーとして即時停止
- unit 不足: 設定画面に明示し、他の再試行を止める
- アドレス不正: そのアドレスだけ失敗

無限再試行は行わない。

## 9. 二重計上を防ぐ集計ルール

DeBank のトークン一覧には、ウォレットに直接あるトークンだけでなく、設定によってプロトコル由来のトークンが含まれる可能性がある。また、DeFi ポジションの構成トークンを個別資産として加算すると、プロトコル評価額との二重計上が起きる。

そのため、次のルールを採用する。

1. HTML 取込では、画面に表示された総額を基準値とする。公式 API では `total_balance.total_usd_value` を基準値とする。
2. HTML 取込では画面のチェーン内訳を使い、公式 API では `total_balance.chain_list[].usd_value` を使う。
3. ウォレット残高の詳細は `all_token_list` のうちウォレット保有と判定できる項目を使用する。
4. DeFi は `all_complex_protocol_list` の `net_usd_value` をポジション評価額として扱う。
5. DeFi 内の supply token、reward token、borrow token は内訳表示に使うが、もう一度総額へ加えない。
6. アプリ独自の明細合計と `total_balance` の差額を毎回計算する。
7. 差が許容値を超えた場合は「内訳不一致」と表示し、総額を勝手に補正しない。

初期許容差の例:

```text
max(1 USD, total_balance × 0.5%)
```

価格更新時刻のずれでも差が出るため、完全一致は要求しない。

## 10. データモデル

### 10.1 アドレス設定

秘密情報を含まない `wallets.json` を使用する。

```json
{
  "schema_version": 1,
  "wallets": [
    {
      "wallet_id": "wallet_main",
      "name": "メインウォレット",
      "address": "0x0000000000000000000000000000000000000000",
      "groups": ["personal"],
      "enabled": true
    }
  ]
}
```

同じアドレスの重複登録を防ぐ。EVM アドレスは保存前に形式を検証し、比較用には小文字正規化値も保持する。

### 10.2 保存ファイル

```text
data/
  config/
    wallets.json
    settings.json
  snapshots/
    2026/
      07/
        runs.jsonl
        wallets.jsonl
        positions.jsonl
  raw/
    2026/07/<run-id>/<wallet-id>/
  backups/
```

raw API response はデバッグに便利だが容量が増えるため、初期設定では保存しないか、直近数回だけ保存する。

### 10.3 取得実行レコード

```json
{"schema_version":1,"record_type":"run","run_id":"run_01...","started_at":"2026-07-11T08:00:00Z","completed_at":"2026-07-11T08:00:21Z","status":"partial","wallet_count":3,"success_count":2,"failed_count":1,"source":"debank_openapi"}
```

### 10.4 ウォレット集計レコード

```json
{"schema_version":1,"record_type":"wallet_snapshot","run_id":"run_01...","wallet_id":"wallet_main","address":"0x...","observed_at":"2026-07-11T08:00:05Z","fetched_at":"2026-07-11T08:00:09Z","total_usd":"12345.67","chain_values":[{"chain_id":"eth","usd_value":"8000.00"},{"chain_id":"arb","usd_value":"4345.67"}],"status":"success","source":"debank_html_clipboard","parser_version":"1.0.0","input_sha256":"..."}
```

### 10.5 資産明細レコード

```json
{"schema_version":1,"record_type":"position_snapshot","run_id":"run_01...","wallet_id":"wallet_main","address":"0x...","chain_id":"arb","location_type":"wallet","protocol_id":null,"asset_id":"arb:0xaf88...","symbol":"USDC","name":"USD Coin","amount":"1200.50","unit_price_usd":"1.00","value_usd":"1200.50","is_verified":true,"is_wallet":true,"is_core":true,"price_status":"priced","fetched_at":"2026-07-11T08:00:09Z","source":"debank_openapi"}
```

数量と金額は浮動小数点の誤差を避けるため、JSON 上では 10 進文字列として保存する。

### 10.6 DeFi ポジション

```json
{"schema_version":1,"record_type":"defi_position_snapshot","run_id":"run_01...","wallet_id":"wallet_main","chain_id":"eth","protocol_id":"compound","protocol_name":"Compound","position_name":"Lending","asset_usd":"5000.00","debt_usd":"1000.00","net_usd":"4000.00","components":[{"role":"supply","symbol":"USDC","amount":"5000"},{"role":"borrow","symbol":"USDT","amount":"1000"}],"fetched_at":"2026-07-11T08:00:09Z","source":"debank_openapi"}
```

`components` は説明用であり、総額へ再加算しない。

## 11. JPY 表示

DeBank の評価額は主に USD を基準としているため、MVP の正規データは USD を保持する。

JPY 表示は次のいずれかとする。

1. 設定画面で利用者が USD/JPY を手動入力する。
2. 無料または許諾された為替 API を一つ追加する。

MVP では手動レートでも目的を満たす。JPY 値だけ保存せず、使用した USD/JPY レートと基準時刻を保存する。

## 12. 過去データ

初期版の「過去データ」は、本アプリが過去に取得して JSONL に保存したスナップショットだけを意味する。

- 今日初めて登録したアドレスについて、1 年前の正確な DeFi 状態を復元しない。
- 過去価格を使った再評価も MVP では行わない。
- 取得していない日の値は存在しないものとして扱う。
- 日付選択時は、その日の取得回を一覧表示する。

将来必要なら、履歴 API、過去ブロック、過去価格を組み合わせた再構築を別機能として追加する。

## 13. セキュリティ

- 秘密鍵とシードフレーズを扱わない。
- MetaMask の接続・署名を要求しない。
- DeBank AccessKey は JSONL、`wallets.json`、ログへ保存しない。
- AccessKey は macOS Keychain を第一候補とする。
- 環境変数を使う場合も `.env` を Git 管理しない。
- クリップボードは利用者が取込ボタンを押した瞬間だけ読み、常時監視しない。
- コピーされた HTML はコードとして実行せず、外部リソースも読み込まない。
- 生 HTML は初期設定では保存せず、解析結果と入力ハッシュだけを残す。
- ローカル API は `127.0.0.1` のみで待ち受ける。
- アドレス一覧も資産総額と結び付く個人情報として扱う。
- データディレクトリ全体を Git 管理対象外にする。
- ログには API の Authorization / AccessKey ヘッダーを出さない。

## 14. エラー表示

利用者向けの表示は簡潔にする。

| 状態 | 表示例 |
|---|---|
| 成功 | `取得完了 12:34` |
| 部分成功 | `DeFi 内訳の一部を取得できませんでした` |
| API キー不正 | `DeBank AccessKeyを確認してください` |
| unit 不足 | `DeBank APIの利用枠が不足しています` |
| レート制限 | `混雑しています。しばらくして再実行してください` |
| ネットワーク | `DeBankへ接続できませんでした` |
| 内訳不一致 | `総額と明細に差があります。DeBankで確認してください` |

前回成功値を表示する場合は、最新値のように見せず `前回値・取得日時` を必ず添える。

## 16. MVP 受入基準

- 公開 EVM アドレスを複数登録できる。
- 対象アドレスの DeBank ページを起動済み Brave に開ける。
- 利用者がコピーした HTML を明示操作でクリップボードから取り込める。
- HTML から対象アドレスを検出し、登録アドレスとの不一致を防止できる。
- 解析結果を保存前に確認できる。
- 必須項目を解析できない HTML を推測で確定保存しない。
- 総資産、アドレス別、チェーン別、トークン別を表示できる。
- DeFi の資産・負債・純資産を表示できる。
- DeFi 構成トークンを二重計上しない。
- 取得結果が JSONL に追記される。
- 保存済みの過去取得回を選べる。
- AccessKey が JSONL やログに残らない。
- 秘密鍵やシードフレーズを一切要求しない。
- 拡張ボタンの自動押下、座標クリック、ページ自動巡回を行わない。
- クリップボードを常時監視しない。

## 17. 実装開始前に必要な情報

最低限、次だけ決めれば実装を開始できる。

1. 登録予定の EVM アドレス数
2. 各アドレスの表示名
3. 拡張機能がコピーする HTML の実サンプル
4. HTML に総額、チェーン、トークン、DeFi 内訳がすべて含まれるか
5. DeFi ポジションを含めるか（本設計では含める想定）
6. JPY 表示は手動為替レートでよいか
7. データ保存先をプロジェクト配下にするか、ユーザーデータディレクトリにするか
8. 公式 API モードも残すか

実アドレスや AccessKey を設計書へ記載する必要はない。実装時も AccessKey はチャットやソースコードへ貼り付けず、ローカル設定画面または Keychain から登録する。

## 18. 将来の代替案

DeBank API の費用や条件が個人利用に合わない場合は、次の優先順位で置き換える。

1. 別の正規ポートフォリオ API を比較する。
2. 対象チェーンを絞り、RPC / インデクサーから直接取得する。
3. DeFi を対象外にし、ネイティブ通貨と指定トークンだけ RPC で取得する。
4. DeBank 画面を利用者が目視し、合計だけ手動スナップショットとして入力する。

RPC だけで未知の全トークンと全 DeFi ポジションを自動発見する実装は、DeBank API 利用より大幅に複雑になる。個人用 MVP では、公式 DeBank API の実利用コストを最初に小さく検証するのが合理的である。

## 19. DeBank HTML 解析結果

検証用HTMLは、JavaScript の初期データではなく、表示後の DOM を HTML 化した形式だった。次の項目を抽出できることを確認した。

| 項目 | 主な検出方法 | 判定 |
|---|---|---:|
| 対象アドレス | `HeaderInfo_address__` を含む class、または EVM アドレス形式 | 取得可能 |
| 総資産 | `HeaderInfo_totalAssetInner__` を含む要素の直接テキスト | 取得可能 |
| 変動率 | `HeaderInfo_changePercent__` を含む要素 | 取得可能 |
| チェーン ID | `AssetsOnChain_item__` の `data-chain` | 取得可能 |
| チェーン名 | `AssetsOnChain_chainName__` を含む要素 | 取得可能 |
| チェーン評価額 | `AssetsOnChain_usdValue__` を含む要素 | 取得可能 |
| ウォレット総額 | `Wallet` プロジェクトの `.projectTitle-number` | 取得可能 |
| トークン表 | `.db-table-headerItem` と `.db-table-row` | 取得可能 |
| トークンのチェーン・ID | `/token/<chain>/<token-id>` 形式のリンク | 取得可能 |
| DeFi 名 | `ProjectTitle_name__` を含む要素 | 取得可能 |
| DeFi 総額 | DeFi プロジェクト内の `.projectTitle-number` | 取得可能 |
| ポジション種別 | `BookMark_bookmark__` を含む要素 | 取得可能 |
| ポジション明細 | `table_header__` と `table_contentRow__` を含む要素 | 取得可能 |

class 名の末尾はビルドごとに変わる可能性があるため、完全一致ではなく、`HeaderInfo_totalAssetInner__` のような意味を持つ接頭部分で照合する。第一候補が見つからない場合だけ、表見出しやリンク形式を使う。

### 19.1 総資産の注意点

総資産要素の子孫には変動率も含まれている。要素全体の `textContent` を金額として解析すると、総額と変動率が連結される可能性がある。そのため、総資産要素直下のテキストノードから最初の通貨値だけを抽出する。

### 19.2 トークン表の解析

サンプルでは表見出しが次の順序だった。

```text
Token | Price | Amount | USD Value
```

列番号を固定せず、見出し文字列から列の意味を決める。トークンリンクの `/token/<chain>/<token-id>` を、シンボルより優先する資産 ID として使用する。`<$0.01` はゼロへ変換せず、`value_upper_bound_usd = 0.01` と `display_value = "<$0.01"` を保存する。

### 19.3 DeFi ポジションの解析

DeFi はプロトコル単位のコンテナ内に複数パネルを持つ。次の階層を保ったまま保存する。

```text
protocol
  └─ position type
       └─ row
            ├─ pool / asset
            ├─ balance
            └─ USD value
```

トークンリンクからチェーンとトークン ID を取得する。ボタン文字列は操作情報なので資産データには含めない。

### 19.4 取得対象の範囲

チェーン一覧の折りたたみ表示や `Unfold N chains` の内容は、MVPでは取得対象外とする。表示されている総資産、表示中のチェーン情報、ウォレット内トークン表、表示中のDeFiポジションだけを保存する。

したがって、HTML取込時にチェーン一覧の展開を要求したり、未展開を警告したりしない。将来、チェーン別の詳細が必要になった場合は、別途手動展開取込または公式APIモードを追加する。

HTMLをコピーする前に必要なのは、ページの読込みが完了し、対象とするウォレット・トークン・DeFi情報が画面に表示されていることだけである。

### 19.5 サンプルファイルの扱い

サンプルには公開アドレスと資産内容が含まれるため、そのまま Git へ追加しない。パーサーの回帰テスト用 fixture を作る場合は、アドレス、数量、評価額、プロフィール情報を架空値へ置換したコピーを使用する。
