# 暗号資産ダッシュボード UX再設計書

> 取引所APIを含む拡張後の画面・データ設計は [exchange-integration-design.md](exchange-integration-design.md) を正とする。本書はウォレット中心のUX原則と既存画面の改善履歴として参照する。現行実装では、更新画面の個別更新を「対象選択 → 更新」の2ステップに統一し、記録日の入力欄は設けない。

- 文書版: 1.0
- 作成日: 2026-07-11
- 対象: 個人利用・ローカルWebアプリ
- 状態: 参照資料（現行実装と異なる箇所はexchange-integration-design.mdとREADMEを優先）
- 目的: 全ウォレットの資産状況を短時間で正しく把握できる画面へ再設計する

## 1. 設計の結論

このアプリの主目的は「データを取り込むこと」ではなく、「全ウォレットを合算した現在の資産を把握すること」である。したがって、起動直後の画面は総資産、前回比、主要資産、推移、保管場所の順に構成し、ウォレット設定やHTML取り込み操作は主画面から分離する。

画面は次の3つの責務に分ける。

| 画面 | 主目的 | 優先度 |
|---|---|---:|
| ダッシュボード | 全体資産を把握する | 最優先 |
| ウォレット詳細 | 1アドレスの内訳を調べる | 補助 |
| データ更新・設定 | HTML取込、アドレス管理を行う | 運用 |

「取り込み対象」と「表示対象」は同じヘッダーに置かない。取り込み対象は更新画面だけ、表示対象はウォレット詳細の見出し直下だけに置く。

## 2. 現状UIの問題

### 2.1 情報の優先順位が逆転している

- 設定と取込操作が最上部を占有している。
- 最も重要な全体資産の詳細が画面中盤以降にある。
- 「取得済みアドレス」「取得回ID」が総資産と同じ強さで横並びになっている。
- 取得回IDは内部管理情報であり、通常閲覧時の主要KPIではない。

### 2.2 操作対象と表示対象の位置関係が不自然

- ページ最上部の「表示対象」が、最下部の個別詳細を操作している。
- 操作と結果が離れており、何が切り替わったのか認識しづらい。
- 取り込み対象と閲覧対象が並び、誤操作を誘発する。

### 2.3 チャートが判断に使いにくい

- 円グラフの描画領域が小さい。
- 凡例とグラフの対応が弱い。
- 小額資産が多数あると、意味のない細いスライスが増える。
- 総額と資産別内訳の集計範囲が一致しない場合、チャートの信頼性が失われる。

### 2.4 暗号資産のデータモデルが画面に反映されていない

- LidoやAaveは資産ではなくプロトコルである。
- stETH、USDTなどの実体資産と、Lido、Aaveなどの保管・運用場所を分離する必要がある。
- DeFiでは供給資産、担保、借入、報酬、LPポジションを同じ「保有トークン」として単純合算できない場合がある。
- 数量の合算は同一資産識別子・同一単位に限定する必要がある。

## 3. ユーザーが最初に答えを得るべき質問

画面上の優先順位は次の質問順とする。

1. 全体でいくら保有しているか。
2. 前回からどれだけ増減したか。
3. 何の資産をどれだけ持っているか。
4. どのウォレット、チェーン、プロトコルに置かれているか。
5. データはいつ取得され、どのアドレスが古いか。
6. 特定ウォレットの内訳はどうなっているか。
7. 新しいHTMLをどう取り込むか。

## 4. 情報設計

### 4.1 グローバルナビゲーション

画面上部は常に次の3項目だけとする。

- 資産概要
- ウォレット
- データ更新・設定

右上には「最終更新 2026/07/11 20:15」のような状態表示と「データを更新」ボタンを置く。取得回IDは履歴画面の詳細情報へ移す。

### 4.2 ダッシュボードの表示順

```text
┌────────────────────────────────────────────────────────────┐
│ 暗号資産ポートフォリオ      最終更新 7/11 20:15 [データ更新] │
├────────────────────────────────────────────────────────────┤
│ 総資産 $220,856.42   前回比 +$3,210 (+1.48%)   3/3 更新済み │
│ ※ 一部ウォレットが古い場合は、ここに警告を表示              │
├───────────────────────────┬────────────────────────────────┤
│ 資産推移（広い折れ線）      │ 資産配分（大きいドーナツ）       │
│ 保存済み記録の推移          │ 上位5資産 + その他              │
├───────────────────────────┴────────────────────────────────┤
│ 保有資産一覧                                                │
│ 資産 / 数量 / 価格 / 評価額 / 比率 / 主な保管場所           │
├────────────────────────────────────────────────────────────┤
│ 保管場所                                                    │
│ ウォレット別 | チェーン別 | プロトコル別                    │
├────────────────────────────────────────────────────────────┤
│ データ品質・更新状態（問題がある時だけ目立たせる）           │
└────────────────────────────────────────────────────────────┘
```

## 5. ファーストビュー

### 5.1 ヘッダー

- 左: アプリ名と現在の画面名。
- 右: 最終更新日時、更新状態、「データを更新」ボタン。
- ウォレット設定フォームは置かない。
- アドレス選択プルダウンは置かない。

### 5.2 総資産サマリー

一つの大きなサマリー領域として扱い、同じサイズのカードを3枚並べない。

主表示:

- 総資産USDを最大の文字で表示。
- 前回取得時からの金額差と比率を隣接表示。
- 比較対象日時を明記。

副表示:

- 更新済みウォレット数 `3 / 3`。
- 最古データの取得日時。
- 未評価資産数。

「最新の取得回ID」は表示しない。技術的なIDはユーザーの資産判断に寄与しない。

### 5.3 データ鮮度

- 全ウォレットが同じ取得回に含まれる場合: `全3アドレス更新済み`。
- 一部が古い場合: `2/3更新済み・1件は2日前`。
- 24時間を超えるデータ: 黄色の注意表示。
- 7日を超えるデータ: 赤色の警告表示。

## 6. 資産配分チャート

### 6.1 表示仕様

- ドーナツ直径はデスクトップで最低280px、推奨320px。
- カードの高さは最低400px。
- ドーナツ中央に合算総額を表示。
- 上位5資産を個別スライス、それ以外を「その他」にまとめる。
- 1 USD未満の資産はチャートと標準一覧から除外する。
- 除外額は `1ドル未満 8件 / 合計 $1.42` のように注記する。
- 色は資産記号に対して固定し、画面間・再読込後も変えない。

### 6.2 凡例

各行に次を表示する。

- 資産記号
- 数量
- USD評価額
- 構成比

例: `stETH 12.3456 | $22,222.22 | 61.5%`

### 6.3 集計整合性

チャート対象額、資産一覧合計、総資産の差額を検証する。

```text
総資産 = 特定できた資産評価額 + 資産不明評価額 + 丸め差
```

差額が1 USD以上または総資産の0.1%以上ある場合、チャート下に警告を表示する。プロトコル名を資産名の代用にして差額を隠してはならない。

## 7. 保有資産一覧

全体資産を確認する主コンポーネントとして、チャート直下に配置する。

| 列 | 内容 |
|---|---|
| 資産 | stETH、ETH、USDT等。可能なら名称も併記 |
| 数量 | 全ウォレット・全ポジションの合計数量 |
| 参考価格 | DeBank表示価格。価格がない場合は `—` |
| 評価額 | 合計USD評価額 |
| 比率 | 総資産に対する構成比と横棒 |
| 保管場所 | Wallet、Lido、Aave等の主な所在 |

標準状態では評価額1 USD以上のみ表示する。「1ドル未満も表示」を有効にすると小額資産を展開する。検索、評価額順、資産名順の並び替えを提供する。

### 7.1 資産識別

同じシンボルでも異なるコントラクトを無条件に合算しない。内部キーは次を基本とする。

```text
chain_id + contract_address
```

ネイティブ資産は `chain_id + native` を使う。画面上で複数チェーンのUSDC等をまとめる場合も、内訳を展開できるようにする。

### 7.2 DeFiポジション

画面では次を明確に分離する。

| 概念 | 例 |
|---|---|
| 資産 | stETH、USDT、aUSDC、LP token |
| プロトコル | Lido、Aave V3、Uniswap V3 |
| ポジション種別 | Staked、Supplied、Borrowed、Liquidity、Reward |
| チェーン | Ethereum、Arbitrum等 |
| 評価額 | ポジションのUSD価値 |

借入は負債としてマイナス表示する。供給額と借入額を両方総資産へ加算しない。純資産は `供給・担保・運用資産 + 報酬 - 借入` とする。

## 8. 推移グラフ

- 横幅を優先し、最低高さ320px。
- 表示期間は `1か月 / 3か月 / 全期間`。
- 点にカーソルを合わせると取得日時、総資産、前回比を表示する。
- 日付ラベルは「DeBankの過去残高」ではなく、保存したスナップショットの取得時点を示す。
- その日の全ウォレットが揃っていない場合は破線または警告マークを付ける。
- 欠損ウォレットの直近値を無断で補完しない。

履歴が1件だけなら、無意味な折れ線を描かず「比較には2回以上の取得が必要」と表示する。

## 9. 保管場所の表示

資産一覧とは別に「どこに置いてあるか」を確認できる領域を設ける。タブで切り替える。

### 9.1 ウォレット別

- ウォレット名
- 短縮アドレス
- 評価額
- 全体比率バー
- 最終取得日時
- 更新状態

行をクリックするとウォレット詳細画面へ遷移する。ダッシュボード上で表示対象プルダウンを操作させない。

### 9.2 チェーン別

- チェーン名
- 評価額
- 比率
- 主な資産

### 9.3 プロトコル別

- プロトコル名
- チェーン
- ポジション種別
- 実体資産
- 純評価額

## 10. ウォレット詳細画面

個別詳細は独立画面またはモーダルではなく、URLまたは画面状態として独立させる。

```text
資産概要 > ウォレット > Main Wallet

[← 全体へ戻る]
Main Wallet  0x1234…abcd        [DeBankで確認] [このアドレスを更新]
総額 $36,111   全体比 61.5%   最終更新 7/11 20:15

資産内訳 | チェーン | DeFiポジション | 取得履歴
```

複数ウォレットを切り替える必要がある場合、プルダウンはこの画面のタイトル横に置く。結果表示と操作を隣接させる。

## 11. データ更新画面

HTML取込はダッシュボードから分離する。

### 11.1 更新フロー（現行実装）

1. 更新対象ウォレットを選択する。
2. `DeBankで開く` を押す。
3. HTMLコピー手順を画面内で短く案内する。
4. `クリップボードを解析` を押す。
5. 個別更新ではヘッドレスブラウザが取得・保存まで行う。
6. アドレス一致、総額、資産内訳、解析警告を確認する。
7. 一括更新では対象件数、完了件数、成功・失敗件数、経過秒数を表示する。

解析と保存を一つのボタンで即時確定しない。誤ったアドレスや不完全なHTMLを保存する危険を減らす。

### 11.2 取得日時

- 画面から記録日を入力させない。
- 取得日時は処理側で自動付与し、`captured_at` として保存する。
- 過去時点の残高をDeBankから取得する機能ではないことを、必要な状態説明で明示する。

### 11.3 一括更新

登録済みの有効ウォレットを1件ずつヘッドレスブラウザで取得し、同じ更新セッションとして保存する。ブラウザ終了後、次のウォレット開始までの追加待機は行わない。1件の失敗で全体を中断せず、結果を成功・失敗に分けて表示する。

## 12. ウォレット設定画面

設定フォームを常時展開しない。設定画面では一覧を基本とし、追加・編集時だけフォームを開く。

| ウォレット名 | アドレス | グループ | 状態 | 操作 |
|---|---|---|---|---|
| Main Wallet | 0x1234…abcd | DeFi | 有効 | 編集 |

- 削除は強い赤ボタンを常時表示せず、編集メニュー内に置く。
- 削除前に、保存済み履歴を残すか削除するか明示する。
- アドレス追加時はEVM形式と重複を検証する。
- 公開アドレス以外の秘密情報を入力させない。
- 保存ボタンは変更がある時だけ有効化する。

## 13. 1 USD未満の表示ルール

標準表示では1 USD未満を非表示にする。ただしデータ自体は削除しない。

| 場所 | 1 USD未満の扱い |
|---|---|
| 総資産 | 含める |
| 円グラフ | 個別表示しない。「その他」または注記 |
| 保有資産一覧 | 初期状態で非表示。切替可能 |
| 個別ウォレット詳細 | 初期状態で非表示。切替可能 |
| JSONL | すべて保存 |

これにより視認性を確保しつつ、少額資産の存在と監査可能性を失わない。

## 14. 状態・例外表示

### 14.1 空状態

- ウォレット未登録: 「最初のウォレットを追加」だけを案内。
- 登録済み・未取得: 「データ更新」画面からウォレット更新を案内。
- 履歴1件: 現在値を表示し、推移比較には追加取得が必要と説明。

### 14.2 部分取得

- 総額は取得できたが資産内訳が不足: 総額は表示し、配分チャートには警告。
- DeFi資産名が不明: プロトコル名を資産名として表示しない。`資産不明 $X` と表示。
- 数量が解析不能: 評価額は集計し、数量を `—` とする。
- アドレス不一致: 保存不可。

### 14.3 二重計上防止

- ウォレット直持ちとDeFi預入済み資産がDeBank上で別計上されるかをHTML構造ごとに確認する。
- 最新スナップショットはウォレットごとに1件だけ採用する。
- 同じ資産行をDOM上の複数表示箇所から重複抽出しない。
- 総額と内訳の差額を常に計算する。

## 15. レスポンシブ設計

### 15.1 デスクトップ（960px以上）

- 最大コンテンツ幅1280px。
- 推移と配分を `60:40` で横並び。
- 資産一覧は全幅。

### 15.2 タブレット・狭いウィンドウ

- 推移、配分を縦並び。
- ドーナツと凡例も必要に応じて縦並び。
- テーブルは横スクロール可能にし、列を潰さない。

### 15.3 モバイル

- 総資産を最上部に維持。
- 資産一覧はカード形式または重要列だけを表示。
- ウォレットアドレスは短縮表示し、コピー操作を付ける。

## 16. 視覚設計

- 主役は数値であり、カードの数を増やしすぎない。
- 背景とカードの境界は弱く、警告と主要操作のみ強い色を使う。
- 緑・赤だけに意味を依存せず、`+`、`-`、テキストを併用する。
- 金額は桁を揃え、USDは小数2桁固定、JPYは小数なし、数量は画面上では最大4桁で表示する。
- 0に近い数量は指数表記またはDeBankの小数表現を正規化する。
- 内部IDや生のJSON表現を通常画面に出さない。

## 17. データ構造への要求

DeFiポジションは最低限次の構造で保持する。

```json
{
  "wallet_id": "wallet_x",
  "chain_id": "eth",
  "protocol_id": "lido",
  "protocol_name": "Lido",
  "position_type": "staked",
  "asset_symbol": "stETH",
  "asset_contract": "0x...",
  "quantity": "119.7016",
  "usd_value": "215618.56",
  "liability": false
}
```

プロトコル合計しか取得できない場合は `asset_symbol: null` とし、推測でLidoを資産名へ入れない。既存HTML表示からstETH等を抽出できた場合のみ、抽出根拠とともに資産として扱う。

## 18. 実装前の受入条件

次の条件を満たす設計・データが揃うまでUI実装に入らない。

- 起動直後、スクロールせず総資産、前回比、主要資産、最終更新を確認できる。
- 設定フォームがダッシュボード上部を占有しない。
- 表示対象の操作と個別詳細が同じ領域にある。
- Lido/Aaveを資産名として表示しない。
- stETH/USDT等の数量と評価額を全ウォレットで合算できる。
- 1 USD未満は標準非表示だが、切替で確認できる。
- 配分チャートは上位5資産とその他に整理される。
- 総資産と資産内訳の差額が表示・検証される。
- 古いウォレットデータを最新値のように見せない。
- 記録日が過去残高取得機能ではないことが明確である。
- 借入を資産として加算しない。
- 解析結果を保存前に確認できる。

## 19. 実装順序案

実装を再開する場合は、次の順序とする。

1. 資産、プロトコル、ポジション種別を分離したデータモデルを確定する。
2. サンプルHTMLで直持ち、Lido、Aaveの抽出結果をテストする。
3. 総額と資産内訳の整合性チェックを実装する。
4. 画面を資産概要、ウォレット詳細、データ更新・設定に分割する。
5. ダッシュボードのサマリーと資産一覧を実装する。
6. 大きな推移グラフと配分チャートを実装する。
7. 1 USD未満の表示切替、更新警告、空状態を実装する。
8. 取得セッションと成功・失敗結果表示を実装する。
9. デスクトップと狭い画面で視覚確認する。

この順序により、見た目だけを先に整えて誤った集計を目立たせることを避け、データの正しさを土台にUXを構築する。
